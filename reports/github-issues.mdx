---
title: "GitHub Issues"
description: "Automatic GitHub Issue creation with the complete security report"
---

At the end of every `generate-summary` job run, Repo Security Bot automatically opens a GitHub Issue whose body contains the entire `security-summary.md` report. This means every scan produces a permanent, searchable record directly in your repository's issue tracker.

## How it works

After the summary Markdown file is generated and uploaded as an artifact, the final step of the `generate-summary` job runs:

```bash
gh issue create \
  --title "🔒 Security Summary Report: $(date +'%Y-%m-%d')" \
  --body-file reports/security-summary.md
```

The `gh` CLI uses the `GITHUB_TOKEN` secret for authentication:

```yaml
- name: Create GitHub Issue
  env:
    GH_TOKEN: ${{ secrets.GITHUB_TOKEN }}
  run: |
    gh issue create \
      --title "🔒 Security Summary Report: $(date +'%Y-%m-%d')" \
      --body-file reports/security-summary.md
```

### Issue title

The title is dynamic — `$(date +'%Y-%m-%d')` is evaluated at runtime and inserts the current UTC date. For example, a run on 2025-07-15 produces:

```
🔒 Security Summary Report: 2025-07-15
```

### Issue body

The `--body-file` flag passes the full contents of `reports/security-summary.md` directly as the issue body. This includes all five scanner sections and the `_Generated by Repo Security Bot_` footer.

## Required permissions

The `generate-summary` job explicitly grants `issues: write` permission so that `gh issue create` is allowed:

```yaml
generate-summary:
  permissions:
    contents: read
    issues: write
```

Without `issues: write`, the `gh` CLI call will fail with a permission error.

<Warning>
  A **new issue is created on every scan run** — scheduled daily runs, pushes to `main`, and manual dispatches each produce their own issue. Over time this accumulates open issues. Teams should close resolved issues promptly, and consider adding a triage label workflow to avoid clutter.
</Warning>

## Using issues for security tracking

GitHub Issues provide a lightweight project-management layer on top of the raw scan output. Once an issue is created you can:

- **Assign to team members** — route specific findings to the engineer or team responsible for that system.
- **Label by severity** — create labels such as `severity:high`, `scanner:bandit`, or `status:in-progress` and apply them during triage.
- **Link to fix PRs** — reference the issue number in your pull request description (`Fixes #123`) so GitHub automatically closes the issue when the PR merges.
- **Close when all findings are resolved** — re-run the workflow after fixes are merged; if the new report is clean, close the issue with a comment linking to the passing run.

### Best practices for security tracking

<Steps>
  <Step title="Triage within 24 hours">
    Assign every new security issue to an owner within one business day. Unassigned issues tend to go stale.
  </Step>
  <Step title="Label by scanner and severity">
    Add labels that reflect both the source scanner (`scanner:gitleaks`, `scanner:checkov`, etc.) and the urgency. This makes filtering and dashboarding easier.
  </Step>
  <Step title="Open one tracking issue per finding type">
    If the same class of finding recurs across runs (for example, a recurring Checkov rule failure), open a dedicated long-lived issue and link each daily scan issue to it rather than treating each scan as independent.
  </Step>
  <Step title="Close promptly after verification">
    After deploying a fix, trigger a manual workflow run (`workflow_dispatch`) to confirm the finding no longer appears, then close the issue with a link to the passing run.
  </Step>
</Steps>
