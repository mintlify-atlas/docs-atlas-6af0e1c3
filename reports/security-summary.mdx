---
title: "Security Summary"
description: "The aggregated security-summary.md report generated after every scan"
---

After all five scanner jobs finish, the `generate-summary` job assembles every individual scanner report into a single `security-summary.md` file. This gives you one place to review all findings without downloading each artifact separately.

## How it is generated

The `generate-summary` job declares all five scanner jobs in its `needs:` list, so GitHub Actions only starts it once every upstream job has completed — regardless of whether those jobs passed or failed.

```yaml
generate-summary:
  runs-on: ubuntu-latest
  permissions:
    contents: read
    issues: write
  needs:
    - gitleaks
    - bandit
    - checkov
    - dockle
    - nuclei
```

Once all scanners are done, their artifacts are downloaded into the `reports/` directory using `actions/download-artifact@v4`, and the following shell script builds the summary:

```bash
SUMMARY="reports/security-summary.md"
echo "# 🔒 Repo Security Bot – Full Security Report" > "$SUMMARY"
echo "" >> "$SUMMARY"

# Include each scanner's Markdown if exists
for scanner in gitleaks bandit checkov dockle nuclei; do
  if [ -f reports/${scanner}-report/${scanner}.md ]; then
    echo "## ${scanner^^} Report" >> "$SUMMARY"
    cat reports/${scanner}-report/${scanner}.md >> "$SUMMARY"
    echo "" >> "$SUMMARY"
  fi
done

echo "---" >> "$SUMMARY"
echo "_Generated by Repo Security Bot_" >> "$SUMMARY"
```

The `for` loop iterates over the five scanner names in order — `gitleaks`, `bandit`, `checkov`, `dockle`, `nuclei` — and for each one checks whether its individual Markdown report exists under `reports/<scanner>-report/<scanner>.md`. If the file is present it appends a `##` section header (the scanner name uppercased via `${scanner^^}`) and then concatenates the full Markdown content.

## Report structure

The generated file always follows this structure:

1. **Title header** — `# 🔒 Repo Security Bot – Full Security Report`
2. **GITLEAKS Report** section — secret scan findings
3. **BANDIT Report** section — Python SAST findings
4. **CHECKOV Report** section — IaC misconfiguration findings
5. **DOCKLE Report** section — Docker image findings
6. **NUCLEI Report** section — DAST scan findings
7. **Footer** — a horizontal rule followed by `_Generated by Repo Security Bot_`

## Report file tree

After all artifacts are downloaded and the summary is built, the `reports/` directory looks like this:

```
reports/
├── security-summary.md
├── gitleaks-report/gitleaks.md
├── bandit-report/bandit.md
├── checkov-report/checkov.md
├── dockle-report/dockle.md
└── nuclei-report/nuclei.md
```

<Note>
  The `security-summary` artifact is uploaded at the end of the `generate-summary` job. To access it, go to the **Actions** tab in your repository, open the relevant workflow run, and scroll down to the **Artifacts** section.
</Note>

## Footer

Every report ends with:

```
---
_Generated by Repo Security Bot_
```

This makes it easy to identify machine-generated reports when they appear as GitHub Issue bodies or downloaded files.

<Tip>
  The summary report is the best starting point for security triage. Open it first to get a bird's-eye view of all findings, then drill into the individual scanner artifacts for raw details.
</Tip>
